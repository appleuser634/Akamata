## 1. 現状の振り返り (Audit サマリ)

### 1.1 Cloudflare / VPS 透過性

| 層 | 透過性 | 現状 |
|---|---|---|
| Handler API (`am.Context`) | ★★★★★ | 完全に同一 |
| App ビルダ (`am.App`) | ★★★★★ | 完全に同一 |
| HTTP client | ★★★★★ | API 同一 (実装は分岐) |
| Env vars | ★★★★★ | API 同一 |
| **DB 初期化** | ★★★☆☆ | **`openSqlite`/`openD1` を comptime で選ぶ必要** |
| FCM push | ★★★☆☆ | Workers 側未実装 |
| MQTT | ★☆☆☆☆ | native 専用、Workers では UnsupportedOnTarget |
| WebSocket upgrade | ★★☆☆☆ | Workers 側 DO 経由が未整備、ハンドラ分岐 |
| ファイル I/O | ★☆☆☆☆ | Workers では揮発、抽象なし |
| `serve()` opts | ★☆☆☆☆ | Workers では `opts` 無視 |

**総合**: 「**Handler 本体は透過、初期化 (`main`/`worker.zig`) と外部 I/O 系は環境意識が必要**」が実態。

### 1.2 機能ギャップ (Hono/Express/FastAPI 比)

★★★★★ クリティカル:
- **multipart/form-data** (ファイルアップロード) — todo_2.md 明示要求
- form-urlencoded body
- Cookie 読み書き
- Session ミドルウェア

★★★★ 高:
- `req.ip` / `X-Forwarded-For`
- OAuth2 Authorization Code クライアント
- Rate limit ミドルウェア
- Room/topic broadcast (WS pub/sub)

★★★ 中:
- Streaming response / SSE
- ETag / If-None-Match (304)
- 構造化 JSON ログ + `request_id` 自動付与
- OpenAPI スキーマ自動生成
- typed config loader

### 1.3 Turso/libsql

- 推奨: **Hrana v3 HTTP** を `am.http_client` 経由で叩く方式 (Workers/native 完全共通)
- D1 の 2-pass コルーチン問題が出ない (Hrana は 1 req=1 res で完結)
- 既存 `am.db.Db` の vtable をそのまま再利用、Stmt も互換維持
- 埋め込みレプリカは P3 で別フラグで対応 (libsql-c Rust toolchain 必要)

## 3. API スケッチ

### 3.1 統一された DB ファクトリ

```zig
// 1 行で書ける、両環境で動く
var db = try am.db.open(gpa, "file:./chat.db");
// or
var db = try am.db.open(gpa, "libsql://my-app-org.turso.io?token=" ++ secret);

// Handler はそのまま
fn createRoom(c: *am.Context(App)) !void {
    var stmt = try c.state().db.prepare("INSERT INTO rooms(name) VALUES(?)");
    defer stmt.deinit();
    try stmt.bindAll(.{"general"});
    _ = try stmt.step();
}
```

URL スキーマ解釈:
- `file:path` → SQLite (native)
- `libsql://host?token=…` → Turso Hrana HTTP (native + Workers)
- `https://host` も libsql として扱う (Turso が両方受け付ける)
- `d1://binding` → D1 (Workers、legacy)

Workers で `file:` を渡したらコンパイル時に `@compileError("file: requires native backend")`、または runtime で `error.BackendMismatch`。

### 3.2 ファイルアップロード

```zig
const Upload = struct {
    name: []const u8,
    avatar: am.http.MultipartFile, // { filename, content_type, data }
};

fn uploadAvatar(c: *am.Context(App)) !void {
    const form = try c.req.multipart(Upload);
    if (form.avatar.content_type[0..6] != "image/"[0..6]) {
        return c.json(.{ .error_kind = "bad_image" }, 400);
    }
    // form.avatar.data はリクエスト arena に居る
    try saveAvatar(form.name, form.avatar.data);
    try c.json(.{ .ok = true }, 200);
}
```

ServeOptions に `max_upload_bytes: usize = 50 * 1024 * 1024` を追加 (デフォ 50MB)。超えると 413 Payload Too Large。

### 3.3 Cookie / Session

```zig
// 読み取り
const sid = c.req.cookie("session") orelse return c.redirect("/login", 302);

// 書き込み
try c.setCookie("session", new_sid, .{
    .max_age_secs = 3600,
    .http_only = true,
    .secure = true,
    .same_site = .lax,
    .path = "/",
});

// Session ミドルウェア
var sessions = try am.mw.SessionStore.initMemory(alloc);
_ = try app.useAll(am.mw.session(State, .{
    .secret = "...",
    .cookie_name = "AKID",
    .store = &sessions.store,
}));

fn login(c: *am.Context(App)) !void {
    var sess = c.session();
    try sess.set("user_id", "alice");
    try c.json(.{ .ok = true }, 200);
}
```

### 3.4 Rate limit

```zig
_ = try app.use("/api/*", am.mw.rateLimit(State, .{
    .key_fn = byIp,
    .max_requests = 100,
    .window_secs = 60,
}));

fn byIp(c: *am.Context(State)) []const u8 {
    return c.req.ip() orelse "unknown";
}
```

### 3.5 Structured access log + request_id

```zig
_ = try app.useAll(am.mw.requestId(State));        // UUID を c.user_data に
_ = try app.useAll(am.mw.accessLog(State, .json)); // JSON 1 line per request

// 出力例:
// {"ts":"2026-05-22T12:00:00Z","req_id":"…","method":"GET","path":"/users/42","status":200,"latency_us":123,"ip":"1.2.3.4"}
```

## 5. 主要な技術判断とリスク

| 判断 | 採択 | 理由 |
|---|---|---|
| Turso 接続経路 | **Hrana HTTP via `am.http_client`** | D1 の 2-pass バリアを完全回避、両環境同一コード |
| D1 の扱い | **deprecated に格下げ**、Turso 推奨 | Hrana 同期セマンティクスのほうが Akamata の vtable と相性◎ |
| multipart 上限 | デフォルト 50MB、ServeOptions で変更 | DoS 防御の現実解 |
| Session store | in-memory + `Db` backend のプラグイン式 | スケール時は Turso/D1 をストアに |
| 埋め込みレプリカ | **v0.2 ではスキップ** (P3 で `-Dwith-turso-replica`) | Rust toolchain 依存を default off |
| Workers の MQTT/WS | **未対応のままドキュメント明示** | 現実的に Cloudflare Queues / DO への置換が必要、別マイルストーン |

### リスク

1. **Hrana エラーモデルのマッピング** — JSON `{type:"error",...}` を `DbError` に漏れなくマップする必要。テストベクトル要
2. **行データ寿命** — Hrana レスポンス JSON を arena に持ち続ける必要。Stmt.deinit までは安全
3. **multipart 実装の堅牢性** — boundary パース、ネスト、CRLF、終端マーカー — テストカバレッジ重視
4. **Session ミドルウェアの signed cookie** — HMAC-SHA256 で署名、replay 防御に jti を入れる選択肢
5. **OpenAPI 自動生成は今回見送り** — comptime reflection の作り込みコストが大きい、v0.3 候補
